# 需求文档：企业 AI 工作台一级落地页

## 背景

GitHub issue #3 提出需要为 WeLink 设计用户 landing page。当前团队已完成对 Mattermost 的改造，将其嵌入自研前端框架作为 IM 部分。主流 IM 产品的 landing page 形态各异：飞书/企业微信仍以对话列表为主，AI 能力以 Bot DM 形式存在；钉钉 PC 端为传统工作台、手机端为 One 功能，并推出独立应用"悟空"，允许用户通过自然语言调用组织内应用。

PM 明确提出：将 AI 工作台作为 WeLink 的一级功能落地，使其成为组织成员登录 IM 后的统一门户入口，分发企业算力、智能体、技能、插件、MCP，并通过收集个人级使用数据挖掘 AI 使用场景，为后续迭代提供信息源。

## 问题

1. **入口分散**：企业内的 AI 能力（对话、知识库、Canvas、督办等）缺乏统一入口，员工不知道在哪里使用、如何调用。
2. **能力发现难**：即便接入大模型，员工也不清楚哪些 Agent/skill/插件可用，无法把 AI 能力与实际工作流结合。
3. **数据孤岛**：员工使用 AI 的行为数据散落在各系统，无法沉淀为企业场景挖掘和迭代的输入。
4. **运营抓手弱**：没有统一工作台，难以向领导层展示 AI 投入的业务价值，也难以推动 adoption。

## 用户

### 主要用户：组织内部普通员工
- 登录 IM 后在 landing page 使用 AI 工作台。
- 通过自然语言或可视化入口调用 Agent、skill、插件、MCP。
- 可自主添加/订阅 skill 或 Agent，也可使用企业预配置的通用能力。

### 次要用户：企业/部门管理员
- 预配置通用 skill、Agent、插件、MCP。
- 管理权限、发布推荐应用、查看使用数据。

### 间接受益者：业务负责人 / 决策层
- 通过工作台使用数据挖掘员工 AI 使用场景，指导后续投资和功能迭代。

## 方案

### 产品形态

采用 **"AI 工作台 + 应用网格混合"** 形态作为一级 landing page：

- **顶部 AI 助手栏**：固定在最上方，支持自然语言输入，作为全功能的统一搜索与路由器。用户可以通过自然语言直达具体功能或调用 Agent。
- **应用/技能网格**：AI 助手栏下方展示企业预配置和员工自主订阅的 skill、Agent、插件、MCP 入口，支持分类、搜索、最近使用、推荐。
- **核心场景卡片**：突出展示一期核心场景"督办"，以及后续规划中的请假、报销、审批、查文档、查同事、发起会议等能力入口。
- **Canvas 工作区入口**：提供做 PPT、写文档等 Canvas 能力的快捷入口。

### 一期能力范围

1. **多轮对话**：基于企业统一 model config 的大模型对话能力。
2. **知识库**：企业当前尚未建立独立知识库；若开源 AI 工作台项目内置 RAG/知识库功能，则优先复用其能力。企业提供 embed 模型用于文档向量化，支持上传企业内部文档（PDF、Word、Markdown 等）构建索引。若开源项目无内置知识库，则一期仅提供对话与skill调用，知识库延后。
3. **Canvas 能力**：做 PPT、写文档等生成式内容创作。若开源项目暂不支持 PPT，则一期仅提供其支持的 artifacts 类型（如 Markdown、文档）。无需与企业办公软件集成，生成内容提供下载即可。
4. **核心应用场景：督办**：将已设计好的督办场景以 skill / 网页动态应用形式集成到工作台。督办设计文档见 `designs/supervision-collab-va/`，当前尚未实现，具体集成方式（API / iframe / React 组件）需在技术设计阶段明确。
5. **应用网格**：企业预配置 + 员工自主订阅 skill/Agent。
6. **个人级使用数据收集**：记录用户会话、skill 调用、Canvas 使用等行为数据，保存在企业后台数据库中，用于场景挖掘。

### 规划中能力（后续迭代）

- 请假、报销、审批、查文档、查同事、发起会议等办公场景。
- 更细粒度的权限与治理。
- 数据分析与场景挖掘平台。

### 技术集成

以"统一门户体验"为优先：
- 方案 1：完全嵌入现有 React + Webpack 自研前端，作为 WeLink 内部路由页面。
- 方案 2：独立部署 AI 工作台应用，通过 iframe 嵌入 WeLink。

最终选型在 explore-opensource 阶段根据候选项目的可嵌入性和技术匹配度确定。

> **选型硬约束（政企 / 国产化）**：本项目面向政企客户，**开源/底层能力选型必须为中国境内项目**，满足"国产化"（信创 / 自主可控）要求；许可协议须**支持嵌入 WeLink 自研产品（含企业版 / 高级版功能）**，或**可通过商务合作买断**。据此，LibreChat / Open WebUI / LobeChat 等境外项目在政企交付方向**不适用**，此前以外部开源为主的候选评估需**按此约束转向国产方案**（国产 AI 工作台 / 对话框架）重新评估。

## 验收标准

1. 登录 WeLink 后，用户可在 landing page 看到 AI 工作台一级入口。
2. AI 工作台包含顶部自然语言助手栏和应用/技能网格。
3. 用户可与大模型进行多轮对话，对话基于企业统一 model config。
4. 用户可使用知识库问答功能，答案基于企业内部知识。
5. 用户可进入 Canvas 工作区做 PPT、写文档。
6. 督办场景以 skill 或网页动态应用形式集成，用户可通过自然语言或点击入口进入。
7. 企业预配置 skill/Agent 对员工可见；员工可自主添加/订阅额外 skill/Agent。
8. 用户使用 AI 工作台的行为数据被记录（会话、skill 调用、Canvas 使用等）。
9. 使用数据可被导出或用于基础场景挖掘分析。
10. 探索开源方案并产出选型报告，明确推荐 LibreChat / Open WebUI / 其他中的最佳方案及集成路径。

## 非目标

- 本期不实现复杂的企业级数据脱敏与审计（后续按需补充）。
- 本期不实现请假、报销、审批、查文档、查同事、发起会议等办公场景的具体 skill（列入规划，但不阻塞一期上线）。
- 本期不实现面向业务负责人的数据分析平台（先保证数据采集和基础查询能力）。
- 本期不要求 AI 工作台必须独立域名部署；以统一门户体验优先。

## 开放问题

1. 企业后台数据库类型和访问方式（MySQL / PostgreSQL / 其他），需在技术设计阶段明确。
2. 督办场景现有实现形态（API / iframe / React 组件），需在技术设计阶段明确。
3. 企业员工账号体系与 AI 工作台用户身份的映射方式（单点登录 / 共享 session / 独立账号）。
