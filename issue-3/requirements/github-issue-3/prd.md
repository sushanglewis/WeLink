# PRD：企业 AI 工作台一级落地页

## 1. 产品概述

### 1.1 目标

将 AI 工作台作为 WeLink 的一级 landing page，成为组织成员登录 IM 后的统一 AI 门户，分发企业算力、智能体、技能、插件、MCP，并通过个人级使用数据挖掘 AI 使用场景。

### 1.2 范围

- **包含**：多轮对话、知识库、Canvas、督办场景集成、应用/技能网格、个人级数据采集、开源方案选型。
- **不包含**：请假/报销/审批/查文档/查同事/发起会议等办公 skill 的具体实现（规划中）、数据分析平台、复杂数据脱敏审计。

### 1.3 目标用户

- 主要用户：组织内部普通员工。
- 次要用户：企业/部门管理员。
- 间接受益者：业务负责人、产品/运营。

## 2. 产品形态

### 2.1 整体布局

AI 工作台作为一级页面，采用 **"AI 助手栏 + 应用/技能网格 + 核心场景卡片"** 的混合布局：

- **顶部固定 AI 助手栏**：自然语言输入框，支持语音/文字，作为全局路由器和搜索入口。
- **应用/技能网格**：展示企业预配置和员工订阅的 skill、Agent、插件、MCP，支持分类、搜索、最近使用、推荐。
- **核心场景卡片**：突出督办入口，以及知识库、Canvas 等一级能力入口。

### 2.2 交互流程

1. 用户登录 WeLink，进入 landing page（AI 工作台）。
2. 用户可通过顶部输入框输入自然语言，AI 识别意图并调用对应能力。
3. 用户也可点击应用/技能网格中的入口，直接进入具体能力。
4. 用户可订阅/管理自己的 skill/Agent。

## 3. 功能模块

### 3.1 多轮对话

- 基于企业统一 model config 的大模型对话。
- 支持上下文记忆、重新生成、复制、分享。
- 对话内容计入使用数据。

### 3.2 知识库

- 企业当前尚未建立独立知识库；若开源 AI 工作台项目内置 RAG/知识库功能，则优先复用其能力。
- 企业提供 embed 模型用于文档向量化。
- 支持上传企业内部文档（PDF、Word、Markdown 等）构建索引。
- 答案展示引用来源。
- 支持追问和重新检索。
- 若开源项目无内置知识库能力，则一期仅提供对话与 skill 调用，知识库延后实现。

### 3.3 Canvas

- 提供做 PPT、写文档等生成式内容创作能力。
- 若开源项目暂不支持 PPT，则一期仅提供其支持的 artifacts 类型（如 Markdown、文档）。
- 无需与企业办公软件集成，生成内容提供下载即可。
- 支持在线编辑、版本历史、导出下载。
- Canvas 使用记录计入使用数据。

### 3.4 督办场景

- 将已设计好的督办场景（见 `designs/supervision-collab-va/`）以 skill / 网页动态应用形式集成。
- 当前督办实现尚未完成，具体集成方式（API / iframe / React 组件）需在技术设计阶段明确。
- 用户可通过自然语言（如"查看我的督办事项"）或点击入口进入。
- 督办页面与 AI 工作台视觉风格统一。

### 3.5 应用/技能网格

- 企业预配置通用 skill/Agent。
- 员工自主添加/订阅。
- 分类展示：办公效率、知识管理、内容创作、业务应用等。
- 搜索、最近使用、我的订阅、推荐。

### 3.6 使用数据收集

- 个人级统计：会话记录、skill 调用记录、Canvas 使用记录、搜索关键词、订阅行为。
- 数据保存在企业后台数据库中。
- 本期先不考虑脱敏。
- 数据用于基础统计和场景标签挖掘。

## 4. 非功能需求

### 4.1 性能

- 页面首屏加载时间 < 3s（在常规企业网络环境下）。
- AI 对话首 token 响应时间 < 2s。

### 4.2 可用性

- 支持 PC 端和移动端自适应（根据 WeLink 客户端）。
- 界面语言为中文。

### 4.3 安全与合规

- 统一使用企业提供的 model config，避免私自调用外部模型。
- 个人级数据存储需符合企业数据规范（保留周期、访问权限在开放问题中进一步明确）。

### 4.4 可扩展性

- skill/Agent/MCP 接入采用插件化设计，便于后续扩展。
- 知识库、模型配置可热更新。

## 5. 技术路线

### 5.1 集成方式

以"统一门户体验"为优先，候选方案：
- **嵌入 React 路由**：完全融入现有自研前端。
- **iframe 嵌入**：独立部署 AI 工作台，通过 iframe 嵌到 WeLink 中。

### 5.2 开源方案选型

在 explore-opensource 阶段评估：
- LibreChat
- Open WebUI
- LobeChat / AnythingLLM / Dify 等补充候选

评估维度：业务匹配度、技术匹配度（React 嵌入/iframe）、维护活跃度、文档、部署成本、数据收集能力。

## 6. 成功指标

1. AI 工作台日活跃用户数（DAU）占 WeLink 日活比例。
2. 人均会话数、skill 调用次数。
3. 督办场景通过 AI 工作台触发的次数。
4. 员工自主订阅 skill 的数量分布。
5. 高频使用场景 Top 10（基于使用数据挖掘）。

## 7. 风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 开源方案与 React 嵌入兼容性差 | 高 | 在 explore-opensource 阶段做嵌入可行性评估 |
| 开源项目无内置知识库，需自建 RAG | 中 | 评估候选项目知识库能力；若无则一期降级为对话+skill |
| 督办集成方式不明确 | 中 | 在技术设计阶段明确 API/iframe/React 组件方案 |
| 个人级数据收集引发合规质疑 | 中 | 后续按需补充审计和脱敏；本期先小范围试点 |
| 一期范围膨胀 | 高 | 严格限制本期只做多轮对话、知识库（复用开源）、Canvas（开源支持类型）、督办、应用网格、数据采集 |

## 8. 开放问题

1. 企业后台数据库类型和访问方式（MySQL / PostgreSQL / 其他），需在技术设计阶段明确。
2. 督办场景现有实现形态（API / iframe / React 组件），需在技术设计阶段明确。
3. 企业员工账号体系与 AI 工作台用户身份的映射方式（单点登录 / 共享 session / 独立账号）。
