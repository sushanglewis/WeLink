# 用户场景: issue-12

## 目标用户

| 用户角色 | 描述 | 核心诉求 |
|----------|------|----------|
| 普通员工 | 通过 Mattermost/IM 与数字员工交互的终端用户 | 用自然语言完成日常工作，不需要学习新工具。 |
| 数字员工 | 组织内统一的 AI Agent 身份 | 理解上下文、调用技能、在 Teable 中与人协作。 |
| 业务管理员 | 为团队配置数字员工、分配 skills 的管理者 | 快速为岗位分配合适的 AI 能力。 |
| 开发者/集成者 | 开发 skills、接入业务系统的人 | 有清晰的 skill 开发规范和分发机制。 |
| 产品经理（PM） | 本产品决策者 | 获得清晰的路线决策和可执行的能力优先级。 |

## 主要场景

### 场景一：员工向数字员工提问并获取基于知识库的回答

- **触发**: 员工在 IM 中 @数字员工 提问。
- **流程**:
  1. 数字员工识别用户身份和权限。
  2. 从企业知识库检索用户有权限访问的相关内容。
  3. 结合 LLM 生成回答。
  4. 如果回答涉及敏感操作，请求人工确认。
- **价值**: 数字员工从“聊天机器人”升级为“懂组织的 AI 同事”。
- **来源**: 访谈 00:02:35、raw-insights 洞察四。

### 场景二：业务管理员为岗位分发 skills

- **触发**: 新业务场景需要数字员工具备某项能力（如报销查询、项目报告生成）。
- **流程**:
  1. 管理员在 skill 市场中找到合适的 skill（官方/第三方/自建）。
  2. 将 skill 分配给特定岗位或数字员工分身。
  3. 员工在 IM 中自然语言调用该 skill。
- **价值**: 避免每个数字员工重复建设，实现能力规模化复用。
- **来源**: 访谈 00:03:35、raw-insights 洞察四。

### 场景三：数字员工通过 Teable 与人协作完成任务

- **触发**: 员工让数字员工“整理本周客户反馈”。
- **流程**:
  1. 数字员工调用 skill 读取 Teable 客户记录。
  2. 生成摘要并写入协作文档/Teable。
  3. 在 Teable 中创建任务卡片，分配给相关人员。
  4. 相关人员收到通知并在 Teable 中查看/编辑。
- **价值**: AI 执行 + 人工确认/补充，形成闭环。
- **来源**: 访谈 00:02:35、raw-insights 洞察三。

### 场景四：员工使用协作文档与数字员工共同编辑

- **触发**: 员工让数字员工“帮我写一份项目周报”。
- **流程**:
  1. 数字员工调用协作文档 skill 创建文档。
  2. 员工在协作文档中查看、修改、评论。
  3. 数字员工根据评论继续补充或调整。
  4. 最终文档保存到企业知识库。
- **价值**: AI 生成初稿，人工打磨，产出可沉淀。
- **来源**: 访谈 00:03:35。

### 场景五：通过协作日程安排会议

- **触发**: 员工让数字员工“约一个下周的评审会”。
- **流程**:
  1. 数字员工读取相关人员的日历空闲时间。
  2. 推荐几个时间选项。
  3. 员工确认后，数字员工创建会议并发送邀请。
- **价值**: 减少反复沟通，提高安排效率。
- **来源**: 访谈 00:03:35。

## 边界场景

| 场景 | 说明 | 处理方式 |
|------|------|----------|
| 用户权限不足 | 数字员工检索到用户无权访问的内容 | 按用户权限过滤，不返回越权内容。 |
| 知识库无答案 | 企业知识库中没有相关内容 | 明确告知用户，并记录为知识缺口。 |
| Skill 执行失败 | 外部系统不可用或参数错误 | 数字员工向用户说明，并保留失败日志供开发者排查。 |
| 多 Agent 编排需求 | 用户提出需要多个 Agent 协作的复杂任务 | 短期不直接支持；记录需求，长期评估是否引入 AI 工作台。 |

## 非目标

- 不为最终用户提供独立的 AI 工作台入口（短期）。
- 不替代现有 IM（Mattermost）的聊天功能。
- 不直接面向 C 端或外部用户提供 SaaS 服务。
