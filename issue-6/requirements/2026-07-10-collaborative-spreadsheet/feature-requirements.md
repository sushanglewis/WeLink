# 在线协同维格表（多维表格）选型特性要求说明

**整理时间**：2026-07-10  
**来源**：2026-07-10 人机协同 IM 平台需求讨论会议纪要 + 督办协作场景设计资料  
**面向对象**：研发选型团队  
**目标**：1-2 天内基于本特性要求完成选型可行性评估，确认 Baserow 是否可用或推荐更合适的开源方案。

---

## 1. 目标与范围

### 1.1 目标
为企业 IM 平台新增“在线协同维格表（多维表格）”一级功能，作为：
- 人与数字员工之间的结构化交互介质；
- 督办、项目管理、需求管理、OKR 等多类业务场景的通用数据承载层；
- 降低重复开发成本，让业务人员能够自由搭建自己的业务应用。

### 1.2 范围
- **本期核心场景**：督办协作（主表 + 跟进子表 + 自动提醒 + 权限隔离）。
- **本期聚焦**：选型评估与嵌入 IM 的最小可行方案。
- **不追求**：复刻飞书多维表格的全部细粒度功能，只抓核心框架。

---

## 2. 选型原则

> **选型硬约束（政企 / 国产化，置顶）**：面向政企客户，**选型必须为中国境内项目**，满足"国产化"（信创 / 自主可控）要求；许可协议须**支持嵌入 WeLink 自研产品（含企业版 / 高级版功能）**，或**可通过商务合作买断**。据此，Baserow（荷兰）、NocoDB / Grist / Teable（境外或 AGPL）等在政企方向受限；此前以外部开源为主的选型评估需**按此约束转向国产方案**重新评估。

1. **必须是"维格表/多维表格"**，底层具备数据库特性（字段类型、表关联、不可合并单元格），而非传统电子表格。
2. **必须支持 AI/Agent 交互**：REST API / CLI / MCP / Bot Token 至少具备其一。
3. **必须支持 Webhook / 自动化**：能够基于数据变更触发外部或内部工作流。
4. **必须与 IM 组织架构打通**：SSO 或至少免重复注册，用户身份一致。
5. **必须可自托管**：数据主权可控；核心 P0 能力可通过宽松开源许可或商务买断获得（不依赖不可买断的付费墙）。
6. **1-2 天内给出结论**：若开源方案无法满足，最晚下周中确定是否基于底层框架自研。

## 3. 功能特性要求

### 3.1 数据模型与表结构（P0）

| 编号 | 特性 | 说明 | 验收标准 |
|------|------|------|----------|
| F-001 | 多维表格/维格表形态 | 列有固定字段类型，行是记录；不支持传统表格的合并单元格、自由排版 | 可创建带字段类型的表，字段值受类型约束 |
| F-002 | 多工作空间（Workspace） | 支持 Workspace 作为顶层隔离单位 | 可创建/切换多个 workspace |
| F-003 | 多数据库/应用（Database/Application） | 一个 workspace 下可创建多个 database | 可在一个 workspace 内创建多个 database |
| F-004 | 多表（Table） | 一个 database 下可创建多个 table | 可创建、重命名、删除 table |
| F-005 | 表关联（Link to Table） | 支持一对一、一对多关联，可在表之间引用数据 | 主表记录可关联多条跟进记录；子表可反查主表 |
| F-006 | 多视图（View） | 同一 table 支持 Grid、Kanban、Calendar、Form、Gallery 等多种视图 | 可为同一张表创建至少 Grid + Form 视图 |
| F-007 | 筛选视图（Filtered View） | 可基于字段条件创建筛选视图，并按用户身份过滤 | 经办人打开筛选视图后只看到自己被分配的行 |

**关键字段类型要求**：
- 文本（text）、长文本（long_text）、数字（number）、日期（date）、单选（single_select）、多选（multiple_select）、协作人（collaborator / multiple_collaborators）、创建人（created_by）、创建时间（created_on）、UUID、公式（formula）、关联字段（link_row）。
- **特殊展示要求**：关联字段在主表行内需支持以标签形式聚合展示子表记录的指定字段，并支持点击标签打开子表记录的表单视图。

### 3.2 AI / Agent 交互能力（P0）

| 编号 | 特性 | 说明 | 验收标准 |
|------|------|------|----------|
| F-008 | REST API | 提供标准的增删改查 API，支持按 table/view/row 操作 | 可通过 curl/脚本完成表的 CRUD |
| F-009 | Bot / Service Account Token | 允许为机器人/脚本创建独立令牌，不以个人身份操作 | 可生成一个仅用于脚本调用的 token |
| F-010 | 支持 CLI 或 MCP | **必须**支持命令行调用，或原生提供 MCP Server，或 REST API 可被快速封装为 MCP；便于 Agent 以自然语言调用表格操作 | 通过 CLI/MCP 完成查表、增删改查；REST 方案需在 1 人天内完成 MCP 封装 |
| F-011 | 筛选/查询接口 | API 支持按条件筛选行，减少 Agent 上下文压力 | 可通过 API 按“经办人=张三”查询未填报记录 |
| F-012 | 批量写入 | 支持一次性插入/更新多条记录 | 可在 60 秒内完成百级数据批量导入 |

### 3.3 Webhook 与自动化（P0）

| 编号 | 特性 | 说明 | 验收标准 |
|------|------|------|----------|
| F-013 | 行级事件 Webhook | 当行创建/更新/删除时，可触发外部 HTTP 回调 | 配置 webhook 后，创建行可在 5 秒内收到回调 |
| F-014 | 自定义自动化流程 | 支持基于触发器创建自动化动作（创建记录、发送通知、调用 webhook） | 主表新增记录后自动在子表创建待填报记录 |
| F-015 | 定时任务/延时任务 | 支持延迟执行或周期性任务，或可通过外部 Cron/Celery 触发 | 可配置 T+24h / T+36h 提醒任务 |
| F-016 | 公式与自动计算 | 支持字段公式、自动填充协作人、自动计算状态 | 协作者字段可根据主表“经办人”自动匹配 |

### 3.4 权限与安全（P0 / P1）

| 编号 | 特性 | 优先级 | 说明 | 验收标准 |
|------|------|--------|------|----------|
| F-017 | Workspace 级共享 | P0 | 可邀请成员加入 workspace，按角色区分 | Admin/Member/Viewer 角色可用 |
| F-018 | 行级数据隔离 | P0/P1 | 经办人只能看到自己相关的记录，不能通过删除筛选条件看到全表 | 普通用户删除筛选器后仍无法查看他人数据 |
| F-019 | 操作权限控制 | P0 | 区分更新、删除权限；经办人仅可更新，督办专员可删除 | 配置后经办人删除按钮不可用 |
| F-020 | 二次确认弹窗 | P0 | 删除、批量修改等敏感操作需要二次确认 | 点击删除后弹出确认框 |
| F-021 | 审计日志 | P1 | 记录谁在什么时间做了什么操作 | 可查看行级/操作级日志 |
| F-022 | 字段级权限 | P2 | 可隐藏或只读特定字段 | 可配置敏感字段对普通用户只读 |

> **权限关键风险**：若开源方案只支持 workspace 级权限，无法做到行级隔离，则经办人可能误删筛选条件后看到全表数据，存在隐私泄露风险。选型时必须明确该项是免费还是付费功能。

### 3.5 IM 集成与用户体验（P0）

| 编号 | 特性 | 说明 | 验收标准 |
|------|------|------|----------|
| F-023 | 嵌入 IM 成为一级功能 | 在 IM 左侧导航新增“多维表格”入口，点击后在主内容区打开 | 入口可见，点击后加载表格页面 |
| F-024 | SSO / 组织架构打通 | 用户无需单独注册表格账号，使用 IM 身份登录 | 点击入口后无需再次输入账号密码 |
| F-025 | IM 通知集成 | 数据变更、分配任务、提醒可通过 IM 消息通知用户 | 被分配协作者收到 IM 消息通知 |
| F-026 | 联系人邀请 | 邀请成员时可选 IM 通讯录联系人 | 选择联系人后自动映射为表格用户并发送邀请 |
| F-027 | 独立域名/自托管 | 可部署在企业内网或自有服务器 | 提供可访问的 URL，IM 可嵌入 iframe |

### 3.6 数据导入与导出（P0）

| 编号 | 特性 | 说明 | 验收标准 |
|------|------|------|----------|
| F-028 | Excel/CSV 导入 | 可通过脚本或粘贴方式导入结构化数据 | 导入后字段类型正确，不篡改原始数据 |
| F-029 | 字段映射配置 | 导入时支持配置 Excel 列与表字段的映射 | 修改映射文件后无需改代码即可适配新模板 |
| F-030 | 导入结果反馈 | 返回成功条数、失败行号及原因 | 可定位到具体哪一行导入失败 |
| F-031 | 数据导出 | 支持导出为 Excel/CSV | 可导出筛选后的数据 |

### 3.7 扩展与生态（P1）

| 编号 | 特性 | 说明 | 验收标准 |
|------|------|------|----------|
| F-032 | 应用市场/插件扩展 | 可扩展视图、字段、自动化组件 | 至少提供扩展机制文档 |
| F-033 | 仪表盘/看板 | 支持基于表格数据生成统计看板 | 可创建简单的数据汇总图表 |
| F-034 | 表单视图 | 可用于数据收集，替代部分审批/填报场景 | 可生成分享链接收集填报 |
| F-035 | 移动端适配 | 在移动端可查看和简单编辑 | 手机浏览器打开页面可用 |
| F-041 | 视图分享（带筛选条件） | 可对某字段设置筛选器，生成分享 URL；打开 URL 自动应用该筛选条件 | 复制“我的待办”视图链接给经办人，打开后只显示该经办人的记录 |

### 3.8 AI 交互与高级表格交互（补充）

| 编号 | 特性 | 优先级 | 说明 | 验收标准 |
|------|------|--------|------|----------|
| F-036 | CLI / MCP 原生支持 | P0 | 除内置自动化/AI 应用外，表格必须支持 **CLI 命令** 或 **MCP Server**；若只有 REST API，也必须能在 1 人天内封装为 MCP，供 Agent 调用 | Agent 可通过 MCP/CLI 执行：查表、按条件查询、插入/更新/删除行、列出 workspace/database/table |
| F-037 | 事件捕获与通知 | P0 | 支持捕获表格行级事件（create/update/delete），并通知协作者或外部系统 | 主表新增记录后，关联的协作者收到通知；Webhook 同时触发外部延时任务 |
| F-038 | 协作者列 | P0 | 支持“协作人 / multiple_collaborators”字段类型，可自动匹配系统用户；当协作者被写入时，该用户收到通知 | 主表“经办人”字段值可自动匹配为 Baserow 用户并写入跟进表协作者列；被匹配用户收到 IM/邮件通知 |
| F-039 | 主表行内子表标签展示 | P0 | 主表与子表为一对多关联时，主表行内以**标签**形式展示所有子表记录的某个字段枚举值；点击标签可打开该子表记录的**表单视图**进行查看/编辑 | 主表“跟进记录”列显示多个标签（如“2026-07-10 / 未填报”）；点击标签弹出子表填报表单 |
| F-040 | 行列级 CURD 权限控制 | P0 | 支持基于用户角色的行级、列级增删改查控制；不同角色对同一表的不同行/列拥有不同权限 | 督办专员可删除任意行，经办人只能更新自己被协作者的行且无法查看/编辑敏感列；配置后用户无权限的按钮/字段不显示 |

## 5. 督办场景验收用例（MVP）

本验收用例用于快速验证选型是否能满足当前最紧迫的业务场景。

### 5.1 用例 1：蓝表批量导入主表
1. 督办专员收到上级蓝表 Excel。
2. 通过 Agent 调用脚本或人工粘贴，将数据导入“督办主表”。
3. 系统返回导入成功条数和失败明细。
4. 主表根据“经办人”字段自动匹配系统用户，填充“协作人”字段。

### 5.2 用例 2：自动创建跟进子表
1. 主表新增一条记录。
2. 系统自动在“跟进记录表”创建一条关联记录，三列填报字段为空。
3. 系统自动触发 Webhook，注册 T+24h / T+36h 检查任务。

### 5.3 用例 3：经办人仅可见自己的待办（行列级权限）
1. 经办人登录 IM，点击“多维表格”。
2. 进入主表的“我的待办”筛选视图。
3. 该视图只显示协作人为该经办人的记录；无权限的行和敏感列对该用户不可见。
4. 经办人无法删除筛选条件看到他人数据（后端行级权限控制，不止前端过滤）。

### 5.4 用例 4：24h / 36h 自动提醒
1. 跟进记录创建 24h 后，系统查询未填报记录。
2. 按经办人聚合，发送一条汇总 IM 提醒消息。
3. 36h 仍未填报，发送升级提醒给督办专员，并附带按任务编号筛选的汇总视图 URL。

### 5.5 用例 5：二次确认防止误删
1. 经办人点击删除按钮。
2. 系统弹出二次确认框。
3. 确认后才执行删除；若配置为经办人无删除权限，则删除按钮不可用。

### 5.6 用例 6：主表行内子表标签展示与表单填报
1. 主表某行关联了 3 条跟进记录，每条跟进记录有“汇报日期”和“填报状态”字段。
2. 主表“跟进记录”列以标签形式展示这 3 条记录的“汇报日期+状态”（如“07-10 未填报”）。
3. 经办人点击其中一个标签，弹出该跟进记录的表单视图。
4. 经办人在表单中填写三列必填字段并保存；保存后主表标签状态实时更新。

### 5.7 用例 7：分享带筛选条件的视图 URL
1. 督办专员在主表创建一个筛选视图，条件为“协作人 = 张三”。
2. 系统生成该筛选视图的分享 URL。
3. 督办专员将 URL 通过 IM 发送给张三。
4. 张三点击 URL 后直接进入多维表格，并自动应用“协作人 = 张三”筛选条件，只看到自己被协办的记录。
5. 该 URL 对未授权用户无效（或提示无权限）。

## 7. 推荐候选方案（参考）

基于当前调研，以下方案可作为重点评估对象：

1. **Baserow**
   - 优点：功能完整、维格表形态成熟、API 完善、Webhook 与自动化齐全。
   - 风险：SSO、行级权限、高级权限为企业版付费功能；自托管界面可能有付费提示。

2. **NocoDB**
   - 优点：AGPL 开源、可基于现有数据库、自托管免费、无行数限制。
   - 风险：UI/UX 与 Baserow 有差距；部分高级功能可能也在企业版。

3. **Teable**
   - 优点：PostgreSQL 原生、支持 SQL、社区版自托管免费。
   - 风险：企业版才有 SSO/高级权限/自动化。

4. **Grist**
   - 优点：Apache 2.0、细粒度权限（行/列级）在免费版即有。
   - 风险：界面更偏数据库/公式，维格表体验弱于 Baserow。

5. **底层自研（Frappe / PostgreSQL+PostgREST+Keycloak+Node-RED）**
   - 优点：完全可控、无付费提示、可按需定制权限。
   - 风险：开发周期长、投入大。

> 详细对比可参考：`docs/research/collaborative-spreadsheet-oss-options.md`

## 8. 决策建议

- **首选路径**：在 1-2 天内重点验证 **Baserow 开源核心版** 是否能满足 P0 需求，尤其是：
  - 开源核心是否支持 workspace 级共享；
  - 开源核心是否能通过视图过滤 + 协作人字段实现“经办人只看自己”的效果；
  - 开源核心是否支持 Webhook + 自动化（或需外部 Celery 补充）。
- **若 Baserow 开源核心在权限/SSO/付费提示上存在不可接受的风险**，立即转向 **NocoDB / Grist** 评估。
- **若开源方案均无法满足行级权限和 SSO**，需在下周中前给出 **底层自研方案** 的人力/时间/风险评估，并上报韩总决策。
