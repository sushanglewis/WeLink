## 1. 引言

### 1.1 编写目的
为 WeLink 新增「在线协同多维表格」一级功能提供一份**完整、明确、可验证**的需求基线，使研发能够在不依赖特定产品的前提下理解目标能力并进行架构与实现设计。

### 1.2 项目背景
WeLink 当前以即时通信（IM）为核心，但办公场景需要结构化的数据承载与协同能力。多维表格将作为：
- **人与数字员工（Agent）之间的结构化交互介质**；
- **督办、项目管理、需求管理、OKR 等多类业务场景的通用数据层**；
- **业务人员自助搭建轻量业务应用**的工具，降低重复开发成本。

本期以**督办协作**为锚点场景验证业务价值，优先在公司内部上线。

### 1.3 参照基线说明（重要）
本说明书以 **Baserow 的公开能力作为「功能完备性参照基线」**。文中大量需求项标注「参照：<某能力>」，含义为：

> **所要求的能力应「不低于」该参照基线，但不指定必须采用 Baserow 实现。**

研发可选择任何技术路径（自研 / 基于任意开源二次开发 / 商业采购），只要**满足本说明书的能力与验收标准**即可。参照基线仅用于：
- 帮助研发快速建立对目标能力的直观认知（"大概要做到 Baserow 那样的程度"）；
- 作为功能完整度的下限标尺，避免低估工作量。

> **选型更新（2026-07-15）**：实际选型已确定为 **[Teable](https://github.com/teableio/teable)**（中国境内项目）。Teable 的能力应不低于本说明书以 Baserow 为参照基线所定义的下限。

### 1.4 术语表

| 术语 | 定义 |
|------|------|
| 多维表格 / 维格表 | 列有固定字段类型、行为记录、底层具备数据库特性（字段约束、表关联、不可合并单元格）的结构化数据表，区别于传统自由排版电子表格 |
| 工作空间（Workspace） | 顶层隔离单位，承载多个应用与成员 |
| 应用 / 数据库（Application/Database） | 工作空间下的业务应用容器，含多张数据表 |
| 数据表（Table） | 字段 + 记录的集合 |
| 记录（Record） / 字段（Field） | 行 / 列 |
| 视图（View） | 同一数据表的多种展示与筛选形态（表格、看板、日历、表单、画廊等） |
| 关联（Link） | 表与表之间的引用关系（一对一 / 一对多） |
| 协作人（Collaborator） | 绑定系统用户的字段类型，用于任务分配与权限 |
| Agent / 数字员工 | 通过机器接口（API/CLI/MCP）调用表格能力的自动化程序或智能体 |
| 督办 | 对上级交办事项进行跟踪、催办、反馈的政务/企业管理场景 |
| 经办人 | 具体承办某一督办事项的责任人 |
| 督办专员 | 负责录入、跟踪、催办、汇总的管理角色 |

## 2. 总体描述

### 2.1 产品目标
1. 提供**可嵌入 WeLink 的多维表格一级功能**，作为结构化协同数据的通用承载层。
2. 支撑**人（业务用户）与 Agent（数字员工）双通道**对数据的读写。
3. 以**督办协作**为首个完整闭环场景，验证并沉淀可复用到其他场景的能力。

### 2.2 用户角色（Personas）

| 角色 | 描述 | 核心诉求 |
|------|------|----------|
| 督办专员（管理角色） | 录入交办事项、创建跟进、催办、汇总上报 | 批量导入、自动跟进、催办提醒、全局视图、汇总分享 |
| 经办人（执行角色） | 承办具体事项并按期反馈 | 只看自己的待办、便捷填报、接收提醒 |
| 部门/区级责任人 | 关注本部门事项进展 | 按组织维度查看、只读为主 |
| 管理员 | 维护工作空间、成员、权限、集成 | 组织打通、权限管控、审计 |
| Agent / 数字员工 | 以程序身份读写表格、触发流程 | 稳定机器接口、事件回调、按条件查询 |

### 2.3 产品范围

**本期包含（In Scope）**
- 多维表格核心数据模型（工作空间/应用/表/字段/记录/视图/关联）；
- 多视图与筛选视图、视图分享；
- 机器可调用接口（API，且可被封装为 CLI/MCP）；
- 事件回调（Webhook）与自动化（触发器、定时/延时任务、公式）；
- 行列级权限与数据隔离、敏感操作二次确认、审计；
- 与 WeLink 的单点登录、消息通知、通讯录集成；
- Excel/CSV 导入导出；
- 督办协作 MVP 全闭环（见第 8 章验收用例）。

**本期不包含（Out of Scope）**
- 复刻市面产品的全部细粒度功能（只抓核心框架）；
- 独立移动端 App（移动端仅需浏览器可用）；
- 复杂 BI/数据可视化平台（仅需基础仪表盘）；
- 跨企业的多租户 SaaS 运营能力（本期为单一企业自托管内部使用）。

### 2.4 运行环境与约束
- **必须可自托管**：部署在企业自有环境/内网，数据主权可控、数据不出域。
- **数据安全**：传输加密、可备份、操作可审计。
- **技术中立**：不限定实现语言与框架，但应便于企业现有团队二次开发与运维（容器化部署优先）。

### 2.5 假设与依赖
- WeLink 已有统一用户体系与组织架构，可提供 SSO 与通讯录接口。
- WeLink 具备消息推送能力，可向指定用户发送通知。
- 企业具备运行容器化服务的基础设施。
- 业务数据规模在本期为「单表万级记录、百级并发用户」量级。

## 3. 功能需求

> 说明：每条需求给出「需求陈述（系统应…）」+「验收标准」+「参照基线」。优先级：**P0**=本期必须；**P1**=本期应尽量，可裁剪；**P2**=后续迭代。

### 3.1 数据模型与表结构（P0）

| 编号 | 需求陈述 | 验收标准 | 参照基线 |
|------|----------|----------|----------|
| REQ-DM-01 | 系统应提供「多维表格」形态：列有固定字段类型、行为记录、不支持合并单元格与自由排版 | 可创建带字段类型的表，字段值受类型约束，非法值被拒绝 | Baserow 字段类型约束 |
| REQ-DM-02 | 系统应支持「工作空间」作为顶层隔离单位 | 可创建/切换多个工作空间，空间间数据隔离 | Baserow Workspace |
| REQ-DM-03 | 一个工作空间下应可创建多个应用/数据库 | 可在一个工作空间内创建多个应用 | Baserow Database |
| REQ-DM-04 | 一个应用下应可创建多张数据表 | 可创建、重命名、删除、复制数据表 | Baserow Table |
| REQ-DM-05 | 系统应支持表间关联（一对一、一对多），可跨表引用数据 | 主表记录可关联多条子表记录；子表可反查主表 | Baserow Link to Table |
| REQ-DM-06 | 系统应支持「协作人」字段类型，可绑定一个或多个系统用户 | 协作人字段可取系统真实用户，写入后可触发通知 | Baserow Collaborator |
| REQ-DM-07 | 系统应支持公式字段，可基于其他字段自动计算 | 可定义公式并随源字段变化自动更新 | Baserow Formula |

### 3.2 视图与展示（P0）

| 编号 | 需求陈述 | 验收标准 | 参照基线 |
|------|----------|----------|----------|
| REQ-VW-01 | 同一数据表应支持多种视图，至少包含「表格视图」与「表单视图」 | 可为同一张表创建表格视图与表单视图并切换 | Baserow Grid/Form |
| REQ-VW-02 | 系统应支持「筛选视图」：基于字段条件过滤记录，且过滤在后端生效 | 创建筛选视图后，仅展示符合条件的记录；删除前端筛选不能泄露其余数据 | Baserow Filtered View |
| REQ-VW-03 | 系统应支持「按用户身份过滤」的视图（如"我的待办"） | 经办人打开该视图只看到分配给自己的记录 | Baserow + 协作人过滤 |
| REQ-VW-04 | 关联字段在主表行内应以**标签**形式聚合展示子表记录的指定字段，且点击标签可打开该子表记录的表单视图 | 主表"跟进记录"列展示多个标签（如"07-10 未填报"）；点击标签弹出子表填报表单 | Baserow 关联字段展示（增强） |
| REQ-VW-05 | 系统应支持生成「带筛选条件的视图分享链接」，打开后自动应用该筛选条件 | 复制"我的待办"视图链接发给经办人，打开后只显示其被分配的记录；对未授权用户无效或提示无权限 | Baserow 视图分享（增强） |
| REQ-VW-06 | 系统宜支持看板、日历、画廊等视图（P1） | 可为同一数据创建至少一种非表格视图 | Baserow Kanban/Calendar/Gallery |

### 3.3 字段类型（P0）

系统应至少支持以下字段类型（参照 Baserow 字段体系）：
文本、长文本、数字、日期（精度可配置，支持到小时）、单选、多选、协作人（单/多）、创建人、创建时间、最后修改人、最后修改时间、UUID、公式、关联（Link）。

验收：可为表创建上述各类型字段，字段值受类型约束；日期字段可配置到小时精度。

### 3.4 机器可调用接口：API / CLI / MCP（P0）

> 目标：让 **Agent / 数字员工** 能以程序方式稳定地读写表格。本组是实现无关的「能力」要求，不规定具体协议形态。

| 编号 | 需求陈述 | 验收标准 | 参照基线 |
|------|----------|----------|----------|
| REQ-AI-01 | 系统应提供标准的增删改查机器接口，支持按 应用/表/视图/记录 维度操作 | 可通过脚本完成表与记录的 CRUD | Baserow REST API |
| REQ-AI-02 | 系统应支持「服务账号 / 机器人令牌」，以非个人身份进行机器调用 | 可生成仅用于脚本调用的令牌，且操作可区分于人类用户 | Baserow Token |
| REQ-AI-03 | 系统能力应可被 **Agent 以自然语言/命令方式调用**：原生提供 CLI 或 MCP，或其 API 可在 1 人天内封装为 CLI/MCP | Agent 可执行：列出 应用/表、查表、按条件查询、插入/更新/删除记录 | （能力要求，无直接参照） |
| REQ-AI-04 | 查询接口应支持按条件筛选记录，以降低 Agent 上下文压力 | 可按"协作人=张三"查询未填报记录 | Baserow filter |
| REQ-AI-05 | 应支持批量写入（一次插入/更新多条记录） | 60 秒内完成百级记录批量导入 | Baserow batch |

### 3.5 事件与自动化（P0）

| 编号 | 需求陈述 | 验收标准 | 参照基线 |
|------|----------|----------|----------|
| REQ-AU-01 | 系统应提供「行级事件回调」：记录创建/更新/删除时可触发外部 HTTP 回调 | 配置回调后，创建记录可在 5 秒内收到事件 | Baserow Webhook |
| REQ-AU-02 | 系统应支持「自动化流程」：基于触发器执行动作（创建记录、发送通知、调用回调） | 主表新增记录后可自动在子表创建待填报记录 | Baserow Automation |
| REQ-AU-03 | 系统应支持「定时 / 延时任务」：可延迟或周期性执行（或可被外部调度器触发） | 可配置 T+24h / T+36h 提醒任务并按时触发 | （能力要求；可借助外部调度） |
| REQ-AU-04 | 系统应支持「协作人自动匹配」：当文本字段包含标准化姓名时，自动匹配为系统用户并写入协作人字段；匹配失败时留空且不阻断主流程 | 主表"经办人"为"张三"时，子表协作人自动写入对应用户；匹配不上则留空 | Baserow 自动化/公式（增强） |
| REQ-AU-05 | 自动化应支持「可取消的延时任务」：目标达成（如已填报）后取消后续提醒 | 经办人在 36h 前完成填报后，36h 升级提醒被取消 | （能力要求） |

### 3.6 权限与安全（P0 / P1）

| 编号 | 需求陈述 | 优先级 | 验收标准 | 参照基线 |
|------|----------|--------|----------|----------|
| REQ-PM-01 | 工作空间级成员与角色管理（管理员/成员/只读等） | P0 | 可邀请成员并按角色区分能力 | Baserow 角色 |
| REQ-PM-02 | **行级数据隔离（后端强制）**：用户只能访问被授权的记录，不能通过清除筛选看到全表 | P0 | 普通用户删除筛选器后仍无法查看他人数据 | （Baserow 企业版能力，本期要求必备） |
| REQ-PM-03 | **行列级增删改查权限**：不同角色对同一表的不同行/列拥有不同权限 | P0 | 督办专员可删除任意行；经办人仅可更新自己被分配的行、且敏感列只读/不可见；无权限的按钮/字段不展示 | （Baserow 企业版能力，本期要求必备） |
| REQ-PM-04 | 敏感操作（删除、批量修改）应二次确认 | P0 | 点击删除弹出确认框，确认后才执行 | 通用 |
| REQ-PM-05 | 审计日志：记录谁在何时对何记录做了何操作 | P1 | 可查看记录级/操作级日志 | Baserow Audit Log |
| REQ-PM-06 | 字段级权限：可隐藏或只读特定字段 | P2 | 可配置敏感字段对普通用户只读 | Baserow Field Permission |

> **权限红线**：行级隔离与行列级权限为**本期必备（P0）**，因为这是督办场景"经办人只看自己、敏感信息不泄露"的业务前提。研发在方案选型/设计时，必须确保该能力可用且为后端强制（非仅前端过滤）。

### 3.7 企业 IM 集成与用户体验（P0）

| 编号 | 需求陈述 | 验收标准 |
|------|----------|----------|
| REQ-IM-01 | 在 WeLink 左侧导航新增「多维表格」一级入口，点击后在主内容区打开 | 入口可见，点击后加载表格页面 |
| REQ-IM-02 | 与 WeLink 单点登录打通，用户无需单独注册/再次登录 | 点击入口后无需再次输入账号密码，身份与 IM 一致 |
| REQ-IM-03 | 与 WeLink 组织架构打通，用户/部门数据一致 | 表格用户与 IM 用户映射一致，无需手工维护两套账号 |
| REQ-IM-04 | 数据变更、任务分配、提醒可通过 WeLink 消息触达用户 | 被分配为协作人后，用户收到 IM 消息通知 |
| REQ-IM-05 | 邀请成员时可选择 WeLink 通讯录联系人 | 选择联系人后自动映射为表格用户并发送邀请 |
| REQ-IM-06 | 通知消息应包含：填报入口链接（自动带筛选条件）、截止时间、简明操作步骤；发送者显示为机器人身份 | 经办人点链接直达"我的待办"，消息含截止时间与 1-2 步操作说明，发送者为 Bot |

### 3.8 数据导入与导出（P0）

| 编号 | 需求陈述 | 验收标准 |
|------|----------|----------|
| REQ-IO-01 | 支持 Excel/CSV 导入（脚本或粘贴方式） | 导入后字段类型正确，不篡改原始数据 |
| REQ-IO-02 | 导入支持「字段映射配置」：配置源列与目标字段的对应关系 | 修改映射配置后无需改代码即可适配新模板 |
| REQ-IO-03 | 导入应返回结果反馈：成功条数、失败行号及原因 | 可定位到具体哪一行因何失败 |
| REQ-IO-04 | **导入不容整体失败**：即使部分记录的协作人无法匹配，记录本身也必须成功导入（协作人留空） | 百级数据导入，个别匹配失败不影响其余记录入库 |
| REQ-IO-05 | 支持导出为 Excel/CSV，可导出筛选后的数据 | 可导出当前筛选视图的数据 |

## 4. 逻辑数据需求（业务层，不含实现细节）

> 本章定义**业务实体与字段**，用于研发理解数据语义。**不包含**任何具体数据库字段 ID、表 ID 等实现细节，由研发设计落库。

### 4.1 核心实体关系

```
工作空间 1 ─── n 应用 1 ─── n 数据表 1 ─── n 记录
数据表 1 ─── n 字段
数据表 n ─── n 数据表        （通过"关联"字段形成表间引用）
数据表 1 ─── n 视图
记录   n ─── n 系统用户      （通过"协作人"字段绑定）
```

督办锚点场景包含两张核心表：
- **督办主表**：一条记录 = 一个交办事项；
- **跟进记录表**：一条记录 = 对某事项的一次跟进/填报，与主表一对多关联。

### 4.2 督办主表（业务字段）

| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| 任务编号 | 文本 | 唯一业务编号（如 SJ20260750） | **全局唯一、必填**，用于导入与跟进关联 |
| 记录标题 | 文本 | 事项标题 | — |
| 序号 | 数字 | — | — |
| 交办时间 | 日期 | — | — |
| 完成时限 | 日期 | — | — |
| 预计完成时间 | 日期 | — | 可选 |
| 任务来源 / 议题名称 / 交办事项 / 交办内容 | 文本/长文本 | — | 可选 |
| 责任单位 / 协办单位 | 文本 | — | — |
| 区级责任人 / 部门主要领导 / 部门责任人 / 科室责任人 | 文本 | — | 可选 |
| 经办人及联系方式 | 文本 | 作为协作人自动匹配的来源 | 导入时写入 |
| 经办人账号 | 协作人（多） | 绑定系统用户，可由自动化根据上一字段填充 | 自动 |
| 部门反馈（落实情况/困难问题/下一步计划/进度评价） | 文本/长文本 | 经办人填报 | — |
| 事项类别 / 板块分类 / 督办频率 / 领导批示 | 文本/长文本 | — | 可选 |
| 跟进记录 | 关联 | 反向关联跟进记录表 | 自动 |
| 创建人/创建时间/最后修改人/最后修改时间 | 系统字段 | 自动 | 自动 |

### 4.3 跟进记录表（业务字段）

| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| 任务编号（文本） | 文本 | 便于识别与按编号筛选 | 脚本写入 |
| 关联主表 | 关联 | 指向督办主表对应记录 | 必填 |
| UUID | UUID | 唯一标识 | 自动 |
| 创建时间 | 日期（到小时） | 用于按小时批次聚合提醒 | 自动、小时精度 |
| 部门反馈（落实情况/困难问题/下一步计划） | 文本 | 经办人填报，**"已填报"判定字段** | 经办人填写 |
| 协作人 | 协作人（多） | @ 经办人，由自动化根据主表经办人生成 | 自动 |
| 创建人 | 创建人 | 应为机器/Agent 账号 | 自动 |

### 4.4 关键业务约束
1. **任务编号唯一性**：主表任务编号全局唯一，作为导入与跨表跟进的业务主键。
2. **导入不容失败**：即使经办人无法匹配为系统用户，主表记录也必须成功导入，协作人字段留空。
3. **跟进记录双写**：创建跟进记录时应同时写入"任务编号（文本）"与"关联主表（关联）"，兼顾可读性与关系完整性。
4. **已填报判定**：默认以跟进记录的三个"部门反馈"字段**全部非空**作为"已填报"标准（可配置）。
5. **小时精度**：跟进记录创建时间精度到小时，用于按小时批次聚合提醒、避免重复打扰。

## 5. 外部接口需求（逻辑层）

### 5.1 用户界面
- 嵌入 WeLink 主内容区，视觉与交互与 WeLink 整体一致；
- 提供表格、表单、看板等视图的完整读写界面；
- 关联子表以标签聚合展示，点击标签打开对应记录表单；
- 无权限的字段/按钮对相应用户不展示。

### 5.2 机器接口能力清单（API / CLI / MCP 之一或组合）
- 列出工作空间 / 应用 / 数据表 / 视图；
- 按条件查询记录（支持按协作人、任务编号、填报状态等过滤）；
- 单条/批量 创建、更新、删除记录；
- 以服务账号身份调用，操作可审计、可与人类用户区分。

### 5.3 事件接口（Webhook）
- 支持记录 创建/更新/删除 事件的对外回调；
- 事件载荷应包含：表标识、记录标识、变更前后关键字段、操作者、时间戳；
- 支持按表/事件类型配置回调目标。

### 5.4 与 WeLink 的接口
- **SSO**：WeLink 身份登录，获取表格访问凭证；
- **通知**：向指定用户发送 IM 消息（文本 + 链接）；
- **通讯录**：读取联系人/组织架构，用于成员邀请与用户匹配；
- **用户匹配**：将文本姓名标准化映射为系统用户（用于协作人自动匹配）。

## 7. 约束（业务 / 合规 / 边界）

> **选型硬约束（政企 / 国产化）**：本项目面向政企客户，**开源/底层能力选型必须为中国境内项目**，满足政企采购"国产化"（信创 / 自主可控）要求；其许可协议必须**支持嵌入 WeLink 自研产品（包括企业版 / 高级版功能）**，或**可通过商务合作买断**。据此，Baserow（荷兰）、Mattermost / LibreChat（美国）等境外项目在政企交付方向**不适用**。本 SRS 中以 Baserow 作为"能力参照基线"**仅为功能完备性标尺，不代表选型指向**；开源选型须按此约束重新评估国产方案（详见 `issue-6/docs/research/`）。

1. **合法授权约束**：无论采用何种实现路径，所用底层软件能力必须以**合法方式获得授权**（开源许可合规或商业采购），**不得使用绕过授权的破解手段**。这是公司业务红线，非技术选型细节。
2. **数据主权约束**：全部业务数据自托管于企业环境，不出域；满足等保与数据安全相关要求。
3. **身份一致性约束**：表格系统用户必须与 WeLink 用户体系统一，不允许出现独立的第二套账号体系。
4. **范围约束**：本期只做核心框架与督办 MVP 闭环，不追求复刻商业产品的全部细粒度功能。
5. **可演进约束**：架构应支持从「内部验证」平滑演进到「可对外分发/政企交付」，集成层宜抽象、避免与单一底层产品深度绑定。

## 8. 验收用例（督办协作 MVP）

> 以下用例为**本期上线验收标准**，全部通过方可视为 MVP 达成。用例与《选型特性说明书》第 5 章一致，此处作为正式验收依据。

### UC-1 蓝表批量导入主表
1. 督办专员提供上级蓝表 Excel。
2. 通过 Agent 调用脚本或人工粘贴导入"督办主表"。
3. 系统返回导入成功条数与失败明细。
4. 主表依据"经办人"字段自动匹配系统用户并填充"协作人"字段（匹配失败留空，不影响导入）。

### UC-2 自动创建跟进记录
1. 主表新增一条记录。
2. 系统自动在"跟进记录表"创建一条关联记录，三个填报字段为空。
3. 系统自动触发事件，注册 T+24h / T+36h 检查任务。

### UC-3 经办人仅见自己的待办（行列级权限）
1. 经办人登录 WeLink，点击"多维表格"。
2. 进入主表"我的待办"筛选视图。
3. 仅显示协作人为该经办人的记录；无权限的行与敏感列不可见。
4. 经办人无法通过删除筛选条件看到他人数据（后端强制行级权限）。

### UC-4 24h / 36h 自动提醒
1. 跟进记录创建 24h 后，系统查询未填报记录。
2. 按经办人聚合，发送**一条**汇总 IM 提醒。
3. 36h 仍未填报，向经办人发送升级提醒，并向督办专员发送按任务编号筛选的汇总视图链接。

### UC-5 二次确认防误删
1. 经办人点击删除按钮，系统弹出二次确认。
2. 确认后才执行；若该角色无删除权限，删除按钮不可用。

### UC-6 主表行内子表标签展示与填报
1. 主表某行关联 3 条跟进记录（含"汇报日期""填报状态"）。
2. 主表"跟进记录"列以标签展示各记录的"日期+状态"。
3. 经办人点击标签，弹出该跟进记录的表单视图。
4. 填写必填字段并保存，主表标签状态实时更新。

### UC-7 分享带筛选条件的视图链接
1. 督办专员创建筛选视图（条件"协作人=张三"）并生成分享链接。
2. 通过 IM 发送给张三。
3. 张三打开链接直达多维表格，自动应用筛选，只看到自己的记录。
4. 该链接对未授权用户无效或提示无权限。
