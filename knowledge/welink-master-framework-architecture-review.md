# 架构评审报告：WeLink（龙智协同）产品技术框架

> **评审角色**：架构师  
> **评审对象**：《WeLink（龙智协同）产品技术框架》  
> **评审日期**：2026-07-10  
> **结论**：**方向成立，但有 3 个高风险假设需要 POC 验证；建议按阶段推进，不建议一次性押注。**

---

## 1. 总体评价

### 1.1 值得肯定的 4 个点

| 编号 | 优点 | 说明 |
|---|---|---|
| A1 | **C/S 壳 + 混合模块的顶层设计合理** | 把 Mattermost 从“宿主”降级为“模块”，符合长期解耦目标，也满足中国本地化改造需求。 |
| A2 | **云优先 + 本地增强的策略清醒** | 没有盲目全本地，也没有完全依赖云端，定位准确。 |
| A3 | **以督办协作作为首个闭环场景** | 选择了一个真实、高频、可量化的业务场景做验证，避免平台空转。 |
| A4 | **数据主权与审计作为核心原则** | 对政企客户是强卖点，也指导了后续技术选型。 |

### 1.2 需要警惕的 3 个总体问题

| 编号 | 问题 | 影响 |
|---|---|---|
| C1 | **技术栈异构度过高** | Mattermost（Go/React）、Baserow（Python/Vue）、LibreChat（Node/React）、本地 Agent Runtime（待定）同时存在，运维和人才成本会快速上升。 |
| C2 | **“本地优先”与“云优先”的边界仍模糊** | 文档说本地增强只用于高价值场景，但 AI 工作台被定义为本地模块，容易引发“全本地”预期。 |
| C3 | **路线图压缩了风险验证时间** | Phase 1 和 Phase 2 并行或紧耦合，如果 Mattermost 改造和 Baserow 选型任一失败，整体节奏会被打乱。 |

---

## 2. 关键架构假设与风险（按严重度排序）

### H1：基于 Mattermost Desktop fork 改造为企业工作台壳

**风险等级**：🔴 高

#### 分析

Mattermost Desktop 的设计目标是 **“多个 Mattermost 服务器的聊天客户端”**，而不是通用企业工作台壳。把一级菜单从 Mattermost webapp 抽到本地，本质上是在逆着它的产品形态改造：

- Mattermost Desktop 的左侧导航、通知逻辑、窗口管理都是围绕“服务器/频道”模型；
- 抽出一级菜单意味着要重写大量 UI 和状态管理；
- 未来每次 Mattermost Desktop 升级，都需要合并你们的 fork，维护成本会持续累积；
- Mattermost webapp 与 Mattermost Desktop 的边界会变模糊，调试和排错变复杂。

#### 建议

**不要深度 fork Mattermost Desktop 的业务逻辑，只把它当作“Electron 基础设施层”**：

1. 保留 Mattermost Desktop 的窗口管理、通知、更新、证书、SSO 等基础设施；
2. 将左侧一级菜单、应用路由、模块生命周期管理抽到你们自己的 **App Shell 层**；
3. Mattermost webapp 作为内容区的一个 WebContentsView 存在，不要改它的内部导航；
4. 如果未来 Mattermost Desktop 升级困难，可以平滑迁移到一个全新的 Electron 壳，而业务模块不受影响。

**一句话**：借 Mattermost Desktop 的壳，但不要把它变成你们的业务架构。

---

### H2：AI 工作台作为本地模块，运行本地 Agent Runtime

**风险等级**：🔴 高

#### 分析

本地 Agent Runtime 的业务价值在竞品分析中已经受到挑战。即使技术上可行，以下风险必须正视：

| 风险点 | 说明 |
|---|---|
| **安全面扩大** | 本地 Agent 可以读文件、执行命令、访问本地网络。一旦 Agent prompt 被注入或 MCP Server 被滥用，后果比云端 Agent 严重得多。 |
| **环境依赖复杂** | 不同用户的操作系统、Python/Node 版本、本地模型、防火墙配置差异巨大，会导致“本地能力在某些机器上不可用”。 |
| **更新与回滚困难** | 云端 Agent 可以热更新；本地 Agent 需要随客户端发布，版本碎片化严重。 |
| **用户体验不一致** | 用户在公司电脑和家里电脑、Windows 和 Mac、桌面端和 Web 端看到的能力不同，会增加支持成本。 |
| **真正的数据主权有限** | 如果 Agent 最终调用的是云端大模型，本地 Runtime 只是减少了部分数据外传，prompt 仍然可能离开本机。 |

#### 建议

1. **AI 工作台 UI 先按本地 Electron 视图实现，但 Agent 执行默认走云端**；
2. 本地 Agent Runtime 作为 **可选插件**，由企业管理员开启，普通用户默认不启用；
3. 本地能力必须明确白名单：可读写的目录、可执行的命令、可调用的本地端口；
4. 所有本地操作必须记录审计日志，关键操作需要用户二次确认；
5. 在 POC 阶段，先用 **一个最小本地能力**（例如“读取本地指定目录并总结”）验证安全、更新、兼容性，再扩展。

---

### H3：MCP 作为模块间集成主干

**风险等级**：🟡 中

#### 分析

MCP 是一个很有潜力的协议，但目前还处于早期：

- **安全模型不完整**：MCP Server 通常以用户权限运行，权限隔离、认证、审计需要自己做；
- **生命周期管理复杂**：stdio 类型 MCP Server 需要客户端启动、监控、重启；SSE 类型需要网络可达和 CORS 处理；
- **错误处理与可观测性弱**：标准协议对重试、超时、链路追踪的定义不够完善；
- **企业现有 B/S API 不会天然是 MCP**：需要为每个企业系统写 MCP adapter，工作量不小。

#### 建议

1. **不要一开始就把所有集成都押在 MCP 上**；
2. 对于已有 REST API 的企业服务，优先用 **REST + API Gateway** 集成，降低接入成本；
3. 把 MCP 定位为 **AI Agent 调用工具的协议**，而不是全系统服务总线；
4. 自建一个 **MCP Gateway**，统一处理认证、审计、白名单、限流，不要让本地 Agent 直接调用任意 MCP Server。

---

### H4：Baserow 满足多维表格 P0 需求

**风险等级**：🔴 高

#### 分析

Baserow 在功能完整度上确实最接近 Airtable，但选型文档已经指出：

- **SSO 是企业版付费功能**；
- **行级权限 / 列级权限 / 高级审计是企业版付费功能**；
- 自托管开源版界面可能存在付费提示；
- 前端是 Vue/Nuxt，如果你们需要深度定制 UI，学习成本和 fork 成本都不低。

如果督办场景要求“经办人绝对不能看到他人的督办事项”，而开源版无法做到真正的行级隔离，这会成为 **硬性阻塞**。

#### 建议

1. **在 POC 阶段必须验证**：开源版 Baserow 的 workspace 级共享 + 视图过滤，是否能被业务方接受为“足够安全”；
2. 同步评估 **Grist**（Apache 2.0，行/列级权限免费）和 **NocoDB**（AGPL，无行数限制）作为备选；
3. 明确企业版采购预算和付费点清单，不要让选型在最后一刻因为预算被推翻。

---

### H5：云优先 + 本地增强的产品策略

**风险等级**：🟡 中

#### 分析

这个策略本身是正确的，但执行中容易走样：

- 研发团队可能倾向于把更多能力下沉到本地，因为“本地模块已经做了”；
- 产品经理可能过度承诺本地能力，导致交付压力；
- 用户对“本地模式”和“云模式”的边界不清楚，会产生困惑。

#### 建议

1. 制定明确的 **本地能力启用条件清单**：例如“涉及本地文件”、“必须离线”、“ latency < 500ms”、“数据不能离开本机”；
2. 在产品 UI 上明确标识当前是“本地模式”还是“云模式”；
3. 所有本地能力必须有 **云端降级路径**，避免桌面客户端故障时用户无法工作。

## 3. 架构改进建议

### 3.1 引入统一身份联邦层

目前方案中 Mattermost、Baserow、LibreChat 各自对接 SSO，容易形成身份孤岛。建议：

```
IM IAM / 企业身份源
        │
        ▼
  Keycloak / Authentik（身份联邦）
        │
   ┌────┴────┐
   ▼         ▼
Mattermost  Baserow / LibreChat / 其他模块
```

- Keycloak/Authentik 作为唯一 OIDC/SAML IdP；
- 各模块只对接一个标准身份源；
- 用户离职/调岗时，在联邦层禁用即可全局失效。

### 3.2 定义模块标准契约

建议所有业务模块都遵循同一套接入契约：

| 契约项 | 说明 |
|---|---|
| **生命周期** | `load` / `focus` / `blur` / `unload` / `dispose` |
| **身份注入** | 壳层提供 `token` / `userInfo` / `tenantId` |
| **事件上报** | 模块通过 `window.clientShell.emit(event, payload)` 上报通知、角标、错误 |
| **deeplink** | 支持 `longzhi://module/{name}?params` 被外部唤起 |
| **能力探测** | 模块可查询当前是否支持本地 Agent、本地模型、离线模式 |

这套契约可以让未来替换 Mattermost、Baserow、LibreChat 时影响最小。

### 3.3 本地 Agent Runtime 采用“进程沙箱”模型

不要把本地 Agent Runtime 直接跑在 Electron Main 进程里，建议：

```
Main 进程
    │
    ▼
Agent Supervisor（Node.js）
    │
    ├── pi runtime（Node/Python，有限权限）
    ├── local MCP servers（stdio）
    └── local model proxy（Ollama / vLLM）
```

- Supervisor 负责启动、监控、重启、日志收集；
- pi runtime 以受限用户/受限目录运行；
- 所有本地文件访问通过 Supervisor 中转，并记录审计日志。

### 3.4 数据层建议统一企业主数据

目前数据分散在：

- Mattermost：用户、消息、频道；
- Baserow：督办主表、跟进表；
- LibreChat：会话、消息、Agent 调用；
- 企业后台 MySQL：UsageEvent、用户映射。

建议尽早建立 **企业主数据服务**：

- 统一用户/组织/部门模型；
- 统一权限模型；
- 统一审计事件模型；
- 各模块通过 API/事件订阅同步，而不是各自维护映射表。

## 4. 推荐实施顺序与验证点

### 阶段 1：客户端基座（6-8 周）

**目标**：做一个薄而稳的 C/S 壳，能内嵌 Mattermost，一级菜单可控。

**必须验证**：
- [ ] 本地一级菜单可以稳定切换多个 WebContentsView；
- [ ] Mattermost webapp 在壳内正常运行，通知、下载、文件上传可用；
- [ ] SSO 走 Keycloak/Authentik 联邦，IM 和多维表格可免登；
- [ ] 模块标准契约（生命周期、事件、deeplink）跑通。

### 阶段 2：多维表格选型与嵌入（4-6 周）

**目标**：确定多维表格方案，完成内嵌和 SSO。

**必须验证**：
- [ ] Baserow / Grist / NocoDB 中至少一个满足权限和 SSO 需求；
- [ ] iframe 嵌入后用户体验可接受（加载速度、主题、导航）；
- [ ] 多维表格数据可通过 API 被 IM Bot / Agent 读取和写入。

### 阶段 3：督办协作 MVP（4-6 周）

**目标**：跑通“Excel 导入 → 主表 → 跟进表 → 通知 → 24h/36h 提醒”完整流程。

**必须验证**：
- [ ] 督办专员通过 Mattermost Bot 完成批量导入；
- [ ] 经办人收到通知并进入 Baserow 填报；
- [ ] 未填报记录按规则分级提醒；
- [ ] 业务方认可这个闭环价值。

### 阶段 4：AI 工作台（云端优先，6-8 周）

**目标**：在壳内提供对话、知识库、Artifacts，走云端 Agent。

**必须验证**：
- [ ] LibreChat / Open WebUI 与企业模型网关对接；
- [ ] AI 工作台 UI 在壳内运行流畅；
- [ ] 用户会话和使用数据可写入企业后台；
- [ ] 督办场景可通过 skill/MCP 被 AI 工作台调用。

### 阶段 5：本地 Agent 增强（可选，8-12 周）

**目标**：在明确高价值场景后，启用本地 Agent Runtime。

**必须验证**：
- [ ] 一个具体场景（如本地文件总结并写入多维表格）端到端跑通；
- [ ] 本地 Agent 的权限、沙箱、审计达到企业安全要求；
- [ ] 更新/回滚机制稳定；
- [ ] 用户反馈证明本地能力确实比云端显著更好。

## 5. 需要补充的关键决策

| 决策项 | 当前状态 | 建议 |
|---|---|---|
| 是否采购 Baserow 企业版？ | 未明确 | 在多维表格 POC 后决策，若权限不可接受则转向 Grist/NocoDB |
| 本地 Agent Runtime 技术栈？ | 待定（Node/Python/Rust） | 先用 Node/Python 快速验证，长期关键路径用 Rust |
| 是否强制桌面客户端？ | 倾向强制 | 建议保留 Web 端最小可用能力，避免完全绑定桌面端 |
| 统一身份联邦方案？ | 未明确 | 建议引入 Keycloak 或 Authentik |
| 企业主数据服务？ | 未规划 | 在 Phase 2 开始设计，避免后期数据孤岛 |

## 6. 结论

作为架构师，我对《WeLink（龙智协同）产品技术框架》的评审结论是：

> **方向成立，架构顶层合理，但有 3 个高风险假设（Mattermost Desktop fork 深度改造、AI 工作台本地 Runtime、Baserow 权限）必须在进入大规模研发前通过 POC 验证。建议按阶段推进，优先证明客户端基座、多维表格选型、督办闭环的价值，再决定是否投入本地 Agent Runtime。**

如果这 3 个高风险假设任一不成立，方案需要重大调整：

- Mattermost Desktop 改造过深 → 改做独立 Electron 壳，仅内嵌 Mattermost；
- 本地 Agent Runtime 成本过高或安全不可控 → AI 工作台改为纯云端；
- Baserow 权限不满足 → 转向 Grist / NocoDB / 底层自研。

建议把本评审中的改进建议补充到主框架文档中，并在 POC 阶段设置明确的 go/no-go 检查点。
